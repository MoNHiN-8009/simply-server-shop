Разработай полноценный мод внутриигрового магазина для Minecraft Forge 1.20.1 на Java 17.

Мод устанавливается на сервер и клиент. Сервер является единственным источником истины. Клиент отвечает только за отображение GUI, визуальные вычисления и отправку действий игрока.

Перед написанием кода полностью проанализируй требования. Если существует действительно блокирующая неоднозначность, сначала задай уточняющий вопрос. Не заменяй требования собственными предположениями и не добавляй неуказанные типы товаров или категорий.

==================================================
1. ТЕРМИНЫ
==================================================

В требованиях используются два разных вида команд.

1. Команды самого мода:

/ms
/ms open
/ms balance
/ms buy category <category_id> slot <slot_id>
/ms admin ...

Они регистрируются и реализуются в Java-коде мода.

2. Исполняемые команды товара.

Это строки, которые администратор вручную указывает в настройках товара. Они хранятся в серверном JSON и выполняются сервером после успешной проверки и оплаты покупки.

Пример строки исполняемой команды товара:

give {player} minecraft:diamond 1

Команды самого мода `/ms ...` не хранятся в JSON. В JSON хранятся только настроенные администратором исполняемые команды конкретных товаров.

==================================================
2. КРИТИЧЕСКИЕ ИНВАРИАНТЫ
==================================================

Эти правила имеют высший приоритет. Любая реализация, нарушающая хотя бы одно из них, считается неправильной.

1. Все категории и товары без исключения создаются и настраиваются сервером.

2. Все категории, товары, цены, расположение, оформление и назначенные товарам строки исполняемых команд хранятся исключительно в JSON-файлах на сервере.

3. В Java-коде, enum, клиентских конфигурациях, lang-файлах и клиентских ресурсах не должно быть конкретных предустановленных категорий или товаров.

4. Разрешается создать универсальные модели CategoryDefinition и ProductDefinition, но запрещается создавать конкретные реализации категорий или заранее заданный список категорий.

5. При первом запуске мод создаёт валидную пустую конфигурацию:

"categories": []

Никакие демонстрационные категории или товары не добавляются в рабочий магазин автоматически.

6. Клиент никогда не получает серверные JSON-файлы целиком и не сохраняет содержимое магазина на диск.

7. При каждом открытии магазина клиент обязательно запрашивает актуальное содержимое у сервера.

8. Сервер формирует типизированные DTO из актуального проверенного снимка серверной JSON-конфигурации.

9. GUI строится динамически только из DTO, полученного от сервера. Клиент заранее не знает количество категорий, их названия, порядок, иконки, расположение и товары.

10. После закрытия GUI клиент полностью удаляет из памяти DTO текущего магазина. При следующем открытии данные запрашиваются заново.

11. Обычный DTO магазина не содержит исполняемые команды товаров.

12. Авторизованный администратор может временно получить исполняемые команды в DTO визуального редактора. Эти данные удаляются с клиента после закрытия редактора и никогда не сохраняются клиентом на диск.

13. При нажатии кнопки покупки клиент отправляет от имени игрока только команду:

/ms buy category <category_id> slot <slot_id>

14. Кнопка покупки не содержит и не отправляет цену, баланс или исполняемую команду товара.

15. Серверный обработчик `/ms buy` самостоятельно находит категорию и товар в актуальной серверной конфигурации.

16. Сервер самостоятельно получает исполняемые команды товара из серверного JSON и выполняет их от имени сервера/консоли.

17. Все цены, балансы, проверки, ограничения, транзакции, списания XR и выполнение исполняемых команд контролируются сервером.

18. Клиент может выполнять только визуальные вычисления: масштабирование, расположение элементов относительно окна, прокрутку, hover-эффекты и отрисовку.

19. Клиент не принимает игровых, финансовых или административных решений.

20. JSON хранит конфигурацию и содержимое магазина. SQLite хранит только изменяемые данные: балансы XR, транзакции, историю покупок, лимиты и аудит.

3. ТЕХНОЛОГИИ И АРХИТЕКТУРА
==================================================

Используй:

- Minecraft Forge 1.20.1;
- Java 17;
- Gradle;
- Forge networking;
- Brigadier;
- Gson;
- SQLite;
- Forge Permission API;
- Screen и GuiGraphics Minecraft 1.20.1.

Раздели код минимум на:

- common — общие модели и безопасные DTO;
- server — конфигурация, SQLite, транзакции, команды и валидация;
- client — GUI, визуальный редактор и отрисовка.

Клиентские классы должны быть изолированы. Выделенный сервер не должен загружать Screen, Minecraft, GuiGraphics или другие client-only классы.

Не добавляй зависимости от Bukkit API. Сервер может выполнить любую команду, зарегистрированную в его CommandDispatcher, независимо от того, каким совместимым серверным расширением она была добавлена.

==================================================
4. СЕРВЕРНЫЕ ФАЙЛЫ
==================================================

Используй следующую структуру:

config/xrshop/
  shop.json
  server.json
  messages.json
  backups/

data/xrshop/
  xrshop.db

Назначение файлов:

- shop.json — категории, товары, исполняемые команды товаров и оформление;
- server.json — серверные ограничения, параметры безопасности и SQLite;
- messages.json — изменяемые сообщения интерфейса;
- backups — резервные копии shop.json;
- xrshop.db — балансы XR, транзакции и аудит.

Все перечисленные JSON-файлы являются серверными. Не создавай их копии в клиентской директории config.

shop.json должен содержать:

- schema_version;
- revision;
- общие настройки GUI;
- глобальный стиль;
- список категорий;
- товары внутри категорий;
- исполняемые команды товаров;
- настройки подтверждения покупки;
- звуки и тексты магазина.

При загрузке JSON используй строгую валидацию.

Если новый JSON невалиден:

- не заменяй текущую рабочую конфигурацию;
- продолжай использовать последний корректный снимок;
- выведи точную причину ошибки;
- укажи проблемное поле или объект.

Не читай JSON с диска при каждом открытии магазина или покупке. После загрузки и проверки сервер хранит неизменяемый снимок конфигурации в памяти.

При сохранении используй:

1. Полную проверку черновика.
2. Создание резервной копии.
3. Запись во временный файл.
4. Проверку записанного файла.
5. Атомарную замену shop.json.
6. Создание нового снимка в памяти.
7. Увеличение revision.

==================================================
5. ИДЕНТИФИКАТОРЫ
==================================================

Используй два технических идентификатора:

- category_id — глобально уникальный ID категории;
- slot_id — уникальный ID товара внутри категории.

Пара `(category_id, slot_id)` однозначно определяет товар.

Отдельный product_id не использовать.

Примеры:

category_id: resources
slot_id: diamonds_16

slot_id является строковым ID товара. Это не номер GUI-слота, не индекс массива и не экранная координата.

Визуальная позиция хранится отдельно:

- order;
- page;
- row;
- column.

Технические ID должны соответствовать формату:

[a-z0-9_.-]+

Установи максимальную длину ID.

Отображаемые названия хранятся отдельно и могут быть на русском:

{
  "category_id": "resources",
  "title": "Ресурсы"
}

Команды и автодополнение используют технические ID. GUI показывает title.

==================================================

Слева в GUI расположен вертикальный список категорий.

Список всегда формируется динамически из серверного DTO.

Категория поддерживает:

- category_id;
- title;
- description;
- enabled;
- order;
- permission;
- icon;
- обычный цвет фона;
- цвет фона при наведении;
- цвет активной вкладки;
- цвет текста;
- рамку;
- прозрачность;
- индивидуальные отступы;

Сервер не отправляет игроку категории, к которым у него нет доступа.

Если доступных категорий нет, GUI показывает empty_message из серверной конфигурации.

Клиент не создаёт запасную или стандартную категорию.

==================================================
9. ИСПОЛНЯЕМЫЕ КОМАНДЫ ТОВАРА
==================================================

Администратор может указать товару одну или несколько команд.

Команды:

- находятся в purchase_commands;
- хранятся только в shop.json на сервере;
- сохраняются без начального символа `/`;
- выполняются последовательно;
- выполняются от имени сервера/консоли;
- не отправляются обычному покупателю.

Поддержи серверные заполнители:

- {player}
- {uuid}
- {price}
- {category_id}
- {slot_id}
- {transaction_id}

Все значения заполнителей формируются сервером.

Не используй заполнители, содержащие произвольный текст обычного игрока.

Перед сохранением конфигурации сервер проверяет:

- максимальную длину команды;
- количество команд;
- допустимость заполнителей;
- возможность разбора команды серверным CommandDispatcher после безопасной тестовой подстановки.

Перед фактическим списанием XR сервер повторно проверяет команды после подстановки настоящих значений.

Если команда не разбирается, покупка отклоняется без списания XR.

==================================================
10. ОТКРЫТИЕ МАГАЗИНА
==================================================

Поддержи:

/ms
/ms open

Открытие работает следующим образом:

1. Игрок отправляет серверную команду открытия.
2. Сервер проверяет право minecraftshop.open.
3. Сервер создаёт уникальную сессию магазина.
4. Сервер запоминает UUID игрока, session_id и revision.
5. Сервер берёт актуальный снимок shop.json из памяти.
6. Сервер фильтрует недоступные категории и товары.
7. Сервер создаёт безопасный StoreViewDto.
8. Сервер отправляет DTO клиенту.
9. Клиент открывает GUI и строит его из полученных данных.

StoreViewDto содержит:

- session_id;
- revision;
- отображаемые настройки GUI;
- доступные категории;
- доступные товары;
- отображаемые цены;
- баланс XR.

StoreViewDto не содержит:

- purchase_commands;
- скрытые категории;
- скрытые товары;
- серверные пути;
- содержимое JSON как строку;
- данные SQLite;
- административные настройки.

Если данных много, передавай DTO частями с ограничением размера пакета.

==================================================
11. ЗАКРЫТИЕ МАГАЗИНА
==================================================

После закрытия Screen клиент удаляет:

- StoreViewDto;
- категории;
- товары;
- настройки оформления текущей сессии;
- полученный баланс;
- session_id;
- revision.

Данные не сохраняются в файлы и не используются при следующем открытии.

При отключении от сервера клиент также очищает все данные магазина.

Сервер удаляет сессию магазина:

- при выходе игрока;
- по тайм-ауту;
- при открытии новой сессии;
- при закрытии, если клиент отправил уведомление.

Сервер не должен полагаться только на уведомление клиента о закрытии. У сессии обязательно должен быть серверный тайм-аут.

Пакеты старой сессии игнорируются.

==================================================
14. ПОКУПКА
==================================================

Покупка выполняется командой:

/ms buy category <category_id> slot <slot_id>

Кнопка GUI отправляет эту команду серверу стандартным способом от имени текущего игрока.

Команда содержит только category_id и slot_id.

Она не содержит:

- цену;
- баланс;
- purchase_commands;
- ItemStack для выдачи;
- результат покупки.

Серверный обработчик команды получает ServerPlayer и передаёт запрос в единый PurchaseService.

PurchaseService используется всеми способами покупки. Не дублируй логику покупки в GUI, командах или сетевых обработчиках.

При активной сессии магазина сервер сравнивает её revision с текущей revision.

Если конфигурация изменилась после открытия GUI:

- отклони покупку;
- отправь код STORE_UPDATED;
- обнови или заново открой магазин;
- не списывай XR.

Если игрок вручную использовал `/ms buy` без открытого GUI, покупка выполняется по текущему серверному снимку после всех проверок.

==================================================
15. ПРОВЕРКИ ПОКУПКИ
==================================================

Сервер проверяет:

1. Команду отправил игрок.
2. У игрока есть право minecraftshop.buy.
3. category_id соответствует допустимому формату.
4. slot_id соответствует допустимому формату.
5. Категория существует в актуальном серверном снимке.
6. Категория включена.
7. У игрока есть право на категорию.
8. Товар существует внутри категории.
9. Товар включён.
10. У игрока есть право на товар.
11. Цена корректна.
12. Не превышен лимит покупок.
13. Закончился cooldown.
14. У игрока нет активной незавершённой покупки.
15. Запрос не нарушает rate limit.
16. Баланса XR достаточно.
17. Все purchase_commands корректны.

Цена всегда берётся из актуального серверного снимка. Значение с клиента не принимается.

Добавь серверную блокировку активной покупки игрока. Повторная команда во время обработки текущей покупки не должна создавать вторую транзакцию.

После завершения блокировка снимается в finally или эквивалентном гарантированном механизме.

==================================================
16. ВАЛЮТА XR
==================================================

XR — целочисленная внутриигровая валюта без дробной части.

Используй long с обязательными:

- проверками переполнения;
- запретом отрицательного баланса;
- запретом некорректных сумм;
- настраиваемым максимальным балансом.

Не используй float или double.

SQLite хранит:

- UUID игрока;
- баланс XR;
- начисления;
- списания;
- причины операций;
- источники операций;
- историю покупок;
- состояния транзакций;
- административный аудит;
- лимиты покупок;
- cooldown.

Все изменения баланса выполняются:

- на сервере;
- в SQLite-транзакции;
- параметризованными SQL-запросами;
- с повторной проверкой актуального баланса;
- с защитой от конкурентного списания.

==================================================
17. ТРАНЗАКЦИЯ ПОКУПКИ
==================================================

Используй уникальный server-generated transaction_id.

Состояния:

- CREATED;
- FUNDS_RESERVED;
- EXECUTING;
- COMPLETED;
- REJECTED;
- REFUNDED;
- REVIEW_REQUIRED.

Порядок покупки:

1. Сервер повторно проверяет товар.
2. Сервер проверяет purchase_commands.
3. Сервер создаёт transaction_id.
4. В одной SQLite-транзакции проверяет баланс и списывает XR.
5. Записывает состояние FUNDS_RESERVED.
6. Перед выполнением команды записывает EXECUTING.
7. Выполняет purchase_commands от имени сервера/консоли.
8. Записывает результат каждой команды.
9. При полном успехе устанавливает COMPLETED.
10. Отправляет игроку результат и новый баланс.

Исполняемые команды запускаются не более одного раза.

Если сервер перезапустился с транзакцией EXECUTING:

- не запускай команду повторно;
- переведи операцию в REVIEW_REQUIRED;
- покажи её администратору.

Если команда не начала выполняться, XR можно безопасно вернуть.

Если команда начала выполняться, завершилась исключением или последовательность выполнилась частично:

- не пытайся автоматически повторить команды;
- не обещай автоматический откат внешних действий;
- установи REVIEW_REQUIRED;
- запиши подробный журнал;
- предоставь alert администратору.

18. ВИЗУАЛЬНЫЙ РЕДАКТОР
==================================================

Добавь внутриигровой визуальный редактор:

/ms admin editor

Редактор доступен только с правом minecraftshop.admin.editor.

Редактор является интерфейсом управления серверной конфигурацией. Он не редактирует локальный клиентский JSON.

При открытии редактора:

1. Сервер проверяет права.
2. Сервер создаёт editor_session_id.
3. Сервер создаёт серверный черновик текущего shop.json.
4. Сервер запоминает base_revision.
5. Авторизованному клиенту отправляется типизированный EditorDto.
6. EditorDto может содержать purchase_commands, потому что получатель является проверенным администратором.

Клиент не получает JSON как строку и не отправляет серверу готовый JSON.

Клиент отправляет типизированные операции:

- создать категорию;
- удалить категорию;
- копировать категорию;
- изменить поле категории;
- изменить порядок категории;
- создать товар;
- удалить товар;
- копировать товар;
- переместить товар;
- изменить поле товара;
- добавить команду;
- изменить команду;
- удалить команду;
- изменить оформление;
- запросить валидацию;
- сохранить;
- отменить.

Каждая операция повторно проверяется сервером.

2. Повторно проверь право администратора.
3. Сравни base_revision с текущей revision.
4. При конфликте отклони сохранение без автоматического слияния.
5. Полностью проверь черновик.
6. Создай резервную копию.
7. Атомарно запиши shop.json.
8. Увеличь revision.
9. Замени серверный снимок.
10. Запиши административное действие в аудит.
11. Уведоми открытые клиенты об изменении магазина.

Черновик удаляется:

- после успешного сохранения;
- после отмены;
- при выходе администратора;
- по тайм-ауту.

После закрытия редактора клиент удаляет EditorDto и purchase_commands из памяти.
