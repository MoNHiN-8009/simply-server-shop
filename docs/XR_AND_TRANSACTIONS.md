# XR и транзакции

XR — целочисленная валюта `long` без дробной части. Отрицательный баланс запрещён, арифметика использует проверки переполнения, а `max_balance` ограничивает итог. Все операции выполняются параметризованными SQL-запросами внутри транзакций.

SQLite хранит balances, ledger начислений/списаний, purchases, результаты каждой команды, лимиты/cooldown и административный audit. Конфигурации магазина в БД нет. Схема находится в `src/main/resources/db/migration/V1__initial.sql` и версионируется через `PRAGMA user_version`.

Покупка проходит `CREATED → FUNDS_RESERVED → EXECUTING → COMPLETED`. До списания сервер повторно находит товар в текущем snapshot, проверяет право, revision, rate limit, блокировку UUID, лимит/cooldown, баланс и все команды. Условный SQL update защищает от конкурентного двойного списания.

После `EXECUTING` команды никогда не повторяются автоматически. Исключение, неуспешный callback Minecraft или частичная последовательность переводят запись в `REVIEW_REQUIRED` и сохраняют детали. Числовой результат команды `0` сам по себе не является ошибкой: мод использует штатный флаг успеха `CommandSourceStack.withCallback`. При старте оставшиеся `EXECUTING` также переводятся в `REVIEW_REQUIRED`. Администратор использует `review show`, затем осознанно выбирает `refund` или `resolve`; автоматический откат внешних действий не обещается.

`review show` подсказывает ID последних транзакций, а `review refund` и `review resolve` — ID из `REVIEW_REQUIRED`. Для `reason` доступны готовые русские причины, но произвольный текст разрешён. В выводе `review list` ID кликабелен; экран `show` добавляет кнопки для подстановки команд возврата и решения.

`CurrencyService.externalCredit(source, external_transaction_id, player_uuid, amount, reason)` подготовлен для будущего серверного адаптера сайта. Пара `(source, external_transaction_id)` уникальна, поэтому повтор не начисляет XR. Сейчас HTTP, REST, порт, платежи и секреты намеренно отсутствуют; клиент Minecraft доступа к методу не имеет.
